## LinkedList

Java's `LinkedList` uses a **doubly-linked list** implementation, not a singly-linked list. Each node contains:

- The element value
- A reference to the previous node
- A reference to the next node

This allows for:

- O(1) insertions/deletions at both ends (as a Queue/Deque)
- O(1) insertions/deletions at a known position
- O(n) random access (must traverse from head/tail)

The doubly-linked structure enables bidirectional traversal, which is why `LinkedList` implements both `Queue` and `Deque` interfaces.

## ArrayList

`ArrayList` is backed by a **resizable array**:

- Initial capacity is 10 by default
- When capacity is reached, the array grows by **50%** of its current size
- Growth involves creating a new, larger array and copying elements (expensive operation)
- Amortized insertion is O(1), but occasional resizing is O(n)

Key implementation details:

- Maintains an internal array `elementData`
- Tracks size separately from capacity
- Random access is O(1)
- Insertions/deletions in the middle are O(n) due to shifting elements

## HashMap

`HashMap` uses an array of buckets with linked lists or trees:

- Default initial capacity is 16 buckets
- Default load factor is 0.75 (triggers resize when 75% full)
- Uses power-of-two sizing for efficient modulo operations

Collision resolution:

- Initially buckets use linked lists
- If a bucket exceeds 8 elements (TREEIFY_THRESHOLD), it converts to a Red-Black Tree
- If a tree bucket shrinks below 6 elements (UNTREEIFY_THRESHOLD), it reverts to a linked list

Since Java 8, `HashMap` uses this hybrid approach for performance optimization when there are many collisions.

## HashSet

`HashSet` is actually just a wrapper around `HashMap`:

- It uses a `HashMap` internally where:
    - Set elements are stored as keys in the map
    - All values in the map are a shared dummy object (PRESENT)
- No duplicates because HashMap keys must be unique
- Performance characteristics mirror HashMap

## TreeMap

`TreeMap` is based on a **Red-Black Tree** implementation:

- Self-balancing binary search tree
- Guarantees O(log n) for get, put, remove, containsKey
- Keys are always sorted according to natural ordering or a provided comparator
- No need for hash calculations
- Allows range-based operations like `subMap()`, `headMap()`, `tailMap()`

## TreeSet

Similar to HashSet-HashMap relationship:

- `TreeSet` is a wrapper around `TreeMap`
- Elements are stored as keys in the internal TreeMap
- Elements are always sorted
- Supports efficient range queries

## PriorityQueue

`PriorityQueue` is implemented as a **binary heap**:

- Complete binary tree stored in an array
- Default is min-heap (smallest element at the root)
- O(log n) for insertion and removal of the root element
- O(1) to view the root (peek/element methods)
- Not thread-safe

## ArrayDeque

`ArrayDeque` uses a **resizable circular array**:

- More efficient than LinkedList for queue operations
- Allows efficient additions/removals at both ends
- No capacity restrictions (grows as needed)
- Not thread-safe
- Null elements are prohibited
- Typically faster than Stack and LinkedList when used as a stack or queue

## ConcurrentHashMap

For multithreaded environments:

- Divides the map into segments (partitioned locking)
- Before Java 8: Used separate locks for separate segments
- Since Java 8: Uses a more fine-grained approach with node-level locking
- Provides thread-safe operations without locking the entire map
- Better concurrency than HashTable
- Doesn't allow null keys or values

## CopyOnWriteArrayList

Thread-safe variant of ArrayList:

- Each mutation creates a fresh copy of the underlying array
- Excellent for read-heavy, write-seldom scenarios
- Readers don't need locks (they see a snapshot)
- Expensive for frequent modifications
- Guarantees no ConcurrentModificationException

## LinkedHashMap/LinkedHashSet

These maintain insertion order or access order:

- `LinkedHashMap` extends `HashMap` but adds doubly-linked list to track order
- Can be configured for LRU caching with `removeEldestEntry()`
- Slightly slower than HashMap for insertions/removals due to maintaining order
- O(1) access like regular HashMap

## WeakHashMap

Specialized map where keys are held by weak references:

- Keys can be garbage collected if no strong references exist
- Used for implementing caches where memory is a concern
- Entry disappears when key is garbage collected

## EnumSet/EnumMap

Specialized implementations for enum keys:

- `EnumSet` uses bit vectors for extremely compact and efficient representation
- `EnumMap` uses arrays indexed by enum ordinal values
- Both provide faster performance than general-purpose collections when working with enums
