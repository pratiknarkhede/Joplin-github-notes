#### **What Problem Does It Solve?**

`ReadWriteLock` can still be slow under high contention. **StampedLock** introduces:

- **Optimistic reads**: No locks for reads, with validation checks.
    
- **Better throughput**: Combines features of `ReentrantLock` and `ReadWriteLock`.
    

#### **How It Works**

1.  **Lock Modes**:
    
    - **Write Lock**: Exclusive (like `ReentrantLock`).
        
    - **Read Lock**: Shared (like `ReadWriteLock`).
        
    - **Optimistic Read**: No lock; checks if data changed after read.
        
2.  **Optimistic Reads**:
    
    - `tryOptimisticRead()` returns a stamp (version number).
        
    - After reading, validate using `validate(stamp)`.![9a1b7dbbe462860332a7944c910f1365.png](../../../_resources/9a1b7dbbe462860332a7944c910f1365.png)
        
3.  We can upgrade optimistic read to write locks
    

#### **Pitfalls**

- **No Reentrancy**: Can deadlock if a thread reacquires the same lock.
    
- **Complexity**: Manual validation required for optimistic reads.
