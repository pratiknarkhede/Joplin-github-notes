### **Using Stereotype Annotations to Add Beans**

- **Definition**: Stereotype annotations are used to mark classes that Spring should instantiate and manage as beans.
- **Common Annotations**:
    1.  `@Component`
    2.  `@Service` - Indicates a service component in the business layer.
    3.  `@Repository` - Marks a data access component.
    4.  `@Controller` - Used for Spring MVC controllers.

##### **Steps to Add Beans with Stereotype Annotations**

1.  **Annotate the Class**:
    
    - Apply the appropriate stereotype annotation (e.g., `@Component`) to the class you want Spring to manage.
        
        ```java
        @Component
        public class Parrot {
            private String name;
        
            public String getName() {
                return name;
            }
        
            public void setName(String name) {
                this.name = name;
            }
        }
        
        ```
        
2.  **Enable Component Scanning**:
    
    - Use the `@ComponentScan` annotation in a configuration class to specify the package where Spring should look for annotated classes.
        
        ```java
        @Configuration
        @ComponentScan(basePackages = "com.example")
        public class ProjectConfig {
        }
        
        ```
        
3.  **Access the Bean**:
    
    - Retrieve the bean from the Spring context.
        
        ```java
        var context = new AnnotationConfigApplicationContext(ProjectConfig.class);
        Parrot p = context.getBean(Parrot.class);
        
        ```
        

##### **Advantages**:

- Reduces boilerplate code by avoiding the need to define methods for each bean.
- Provides specialized annotations for better organization (e.g., `@Service` for business logic).

##### **Limitations**:

- Only works for classes you control (i.e., you cannot add external library classes as beans).
- Offers less control over the creation process compared to the `@Bean` annotation.

#### **Adding Beans Programmatically**

Introduced in Spring 5, programmatic registration of beans provides flexibility in dynamic scenarios.

##### **Steps for Programmatic Registration**:

1.  **Create an Instance**:
    
    - Instantiate the object you want to add to the context.
        
        ```java
        Parrot parrot = new Parrot();
        parrot.setName("Kiki");
        
        ```
        
2.  **Define a Supplier**:
    
    - Create a `Supplier` for the object.
        
        ```java
        Supplier<Parrot> parrotSupplier = () -> parrot;
        
        ```
        
3.  **Use `registerBean`**:
    
    - Call the `registerBean` method on the Spring context to add the bean.
        
        ```java
        context.registerBean("parrotBean", Parrot.class, parrotSupplier);
        
        ```
        
4.  **Access the Bean**:
    
    - Retrieve the bean from the context as usual
        
        ```java
        Parrot retrievedParrot = context.getBean(Parrot.class);
        
        ```
        

##### **When to Use**:

- Scenarios requiring dynamic bean registration (e.g., conditional beans based on runtime parameters).

### **Key Takeaways** 

1.  **Stereotype Annotations** (e.g., `@Component`) simplify bean registration by automatically adding annotated classes to the Spring context.
2.  **Programmatic Bean Registration** allows dynamic addition of beans, providing flexibility for runtime conditions.
3.  Each method for bean addition (annotations or programmatic) has specific use cases and trade-offs in terms of control, simplicity, and applicability.

#### **Useful Annotations for Bean Lifecycle**

1.  **@PostConstruct**:
    
    - Executes initialization logic after the bean is created.
    - Example
        
        ```java
        @Component
        public class Parrot {
            @PostConstruct
            public void init() {
                this.name = "Kiki";
            }
        }
        
        ```
        
2.  **@PreDestroy**:
    
    - Executes cleanup logic before the bean is destroyed.
    - Caution: Avoid placing critical operations here, as Spring may fail to call this method under some circumstances.
