### **How `@Transactional` Works Under the Hood (Spring Boot)**

&nbsp;

## **1\. What is `@Transactional`?**

- It’s a Spring annotation that **wraps a method in a database transaction**.
- If the method succeeds → **commits** the transaction.
- If an exception occurs → **rolls back** the transaction.

### **Example**

```java
@Service
public class BankService {

    @Autowired
    private AccountRepository accountRepository;

    @Transactional  
    public void transferMoney(Long fromId, Long toId, double amount) {
        // Deduct from sender
        Account fromAccount = accountRepository.findById(fromId).orElseThrow();
        fromAccount.setBalance(fromAccount.getBalance() - amount);
        accountRepository.save(fromAccount);

        // Add to receiver
        Account toAccount = accountRepository.findById(toId).orElseThrow();
        toAccount.setBalance(toAccount.getBalance() + amount);
        accountRepository.save(toAccount);

        // If anything fails, BOTH changes are rolled back!
    }
}
```

## **2\. How Does Spring Implement `@Transactional`?**

### **A. Proxy-Based Magic (Default Behavior)**

When you annotate a method with `@Transactional`, **Spring creates a proxy** around your object.

#### **Step-by-Step Process**

1.  **Proxy Creation**
    
    - Spring wraps your `BankService` in a **proxy** (like a middleman).
    - This proxy handles transaction logic.
2.  **Before Method Execution**
    
    - Opens a **database transaction** (via `DataSource`).
    - Sets **transaction isolation level** (e.g., `READ_COMMITTED`).
3.  **Method Execution**
    
    - Calls the **real method** (`transferMoney()`).
4.  **After Method Execution**
    
    - If **no exception** → **commits** the transaction.
    - If **exception** → **rolls back** the transaction.

### **B. How Does Spring Decide to Roll Back?**

- By default, **only unchecked exceptions (`RuntimeException`)** trigger rollback.
- Checked exceptions (like `IOException`) **do not rollback** unless specified:
    
    ```java
    @Transactional(rollbackFor = {SQLException.class, IOException.class})
    ```

## **3\. What Happens When You Call `@Transactional` from Another Method?**

### **Case 1: Calling from the Same Class (Problem!)**

```java
@Service
public class BankService {

    public void processPayment() {
        transferMoney(1L, 2L, 100.0);  // ❌ Transaction WON'T work!
    }

    @Transactional  
    public void transferMoney(Long fromId, Long toId, double amount) {
        // ... logic
    }
}
```

#### **Why?**

- **Proxy limitation**: Spring’s proxy only intercepts **external method calls** (from another Spring bean).
- When you call `transferMoney()` **internally**, it bypasses the proxy → **no transaction!**

### **Fix: Self-Injection or Use `TransactionTemplate`**

#### **Option 1: Self-Injection (Hacky but works)**

```java
@Service
public class BankService {

    @Autowired  
    private BankService self;  // Inject a proxy of itself!

    public void processPayment() {
        self.transferMoney(1L, 2L, 100.0);  // ✅ Now transactional!
    }

    @Transactional  
    public void transferMoney(Long fromId, Long toId, double amount) {
        // ... logic
    }
}
```

#### **Option 2: `TransactionTemplate` (Recommended)**

```java
@Service
public class BankService {

    @Autowired  
    private TransactionTemplate transactionTemplate;

    public void processPayment() {
        transactionTemplate.execute(status -> {
            transferMoney(1L, 2L, 100.0);  // ✅ Manual transaction
            return null;
        });
    }

    public void transferMoney(Long fromId, Long toId, double amount) {
        // ... logic
    }
}
```

* * *

## **4\. Key Takeaways**

1.  **`@Transactional` uses proxies** → only works on **external method calls**.
2.  **Internal calls bypass the proxy** → **no transaction!**
3.  **Solutions**:
    - **Self-injection** (not ideal, but quick).
    - **`TransactionTemplate`** (better for programmatic control).
4.  **Always test transactions** (use `@DataJpaTest` or logs to verify).

* * *

### **Interview Question Example**

**Q:** *"If I call an `@Transactional` method from another method in the same class, does the transaction work?"*  
**A:**

> *"No, because Spring’s transaction management relies on proxies. When you call the method internally, it bypasses the proxy, so no transaction is created. To fix this, you can either self-inject the bean or use `TransactionTemplate` for programmatic transactions."*
