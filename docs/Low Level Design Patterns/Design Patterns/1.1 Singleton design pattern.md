### Core Concept

The Singleton pattern is a creational design pattern that ==ensures a class has only one instance and==

==provides a global point of access to that instance.==

&nbsp;

![9f861b8db1f16d90cd966c7a4035205e.png](../../_resources/9f861b8db1f16d90cd966c7a4035205e.png)

&nbsp;

It's classified as a creational pattern because it deals with object creation mechanisms, trying to create objects in a manner suitable to the situation.

The fundamental idea behind the Singleton pattern is to:

1.  **Restrict the instantiation of a class to a single instance.**
2.  **Provide a global point of access to that instance.**

&nbsp;

&nbsp;

## Implementation in Java

Let's explore two common ways to implement the Singleton pattern in Java:

### 1\. Eager Initialization

In eager initialization, the instance of the Singleton class is <span style="color: #000000;">created at the time of class loading.</span>

&nbsp;

```java
public class EagerSingleton {
    private static final EagerSingleton instance = new EagerSingleton();
    
    private EagerSingleton() {}
    
    public static EagerSingleton getInstance() {
        return instance;
    }
}

```

&nbsp;

&nbsp;

### 2\. Lazy Initialization (thread-safe using double-checked locking)

In lazy initialization, the instance is created only when it's first requested.

&nbsp;

```java
public class LazySingleton {
    private static volatile LazySingleton instance;
    
    private LazySingleton() {}
    
    public static LazySingleton getInstance() {
        if (instance == null) {
            synchronized (LazySingleton.class) {
                if (instance == null) {
                    instance = new LazySingleton();
                }
            }
        }
        return instance;
    }
}
```

&nbsp;

### 3\. Initialization-on-demand holder idiom

This method is thread-safe without using synchronization.

&nbsp;

```java
public class Singleton {
    private Singleton() {}
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

&nbsp;

<span style="color: var(--primary-text);">This technique uses a **static inner class (`Holder`) to lazy-load the Singleton instance**. Here's a step-by-step explanation:</span>

1.  <span style="color: var(--primary-text);">**Class Loading**: When `Singleton.getInstance()` is called, the JVM loads the `Singleton` class.</span>
    
2.  <span style="color: var(--primary-text);">**Inner Class Loading**: The JVM then loads the `Holder` class, which is a static inner class of `Singleton`.</span>
    
3.  <span style="color: var(--primary-text);">**Instance Creation**: During `Holder` class loading, the `INSTANCE` field is initialized with a new `Singleton` instance.</span>
    
4.  <span style="color: var(--primary-text);">**Return Instance**: The `getInstance()` method returns the `INSTANCE` field, which now holds the lazily created Singleton instance.</span>
    

<span style="color: var(--primary-text);">Suppose two threads, T1 and T2, call `Singleton.getInstance()` simultaneously:</span>

1.  <span style="color: var(--primary-text);">T1 loads the `Singleton` class.</span>
    
2.  <span style="color: var(--primary-text);">T1 loads the `Holder` class.</span>
    
3.  <span style="color: var(--primary-text);">T1 initializes the `INSTANCE` field with a new `Singleton` instance.</span>
    
4.  <span style="color: var(--primary-text);">T2 loads the `Singleton` class (already loaded by T1).</span>
    
5.  <span style="color: var(--primary-text);">T2 loads the `Holder` class (already loaded by T1).</span>
    
6.  ==<span style="color: var(--primary-text);">T2 returns the existing `INSTANCE` field (created by T1).</span>==
    

<span style="color: var(--primary-text);">Both threads receive the same Singleton instance, ensuring thread-safety.</span>

&nbsp;

<span style="color: var(--primary-text);">**Breaking the Singleton Pattern:**</span>

<span style="color: var(--primary-text);">**Reflection:** Use reflection to create multiple instances.</span>

```java
import java.lang.reflect.Constructor;

public class SingletonBreaker {

    public static void main(String[] args) throws Exception {

        // Get the original Singleton instance
        SingletonClass instance1 = SingletonClass.getInstance();
        System.out.println("Instance 1 hashcode: " + instance1.hashCode());

        // Use reflection to break the Singleton
        Constructor<SingletonClass> constructor = 
            SingletonClass.class.getDeclaredConstructor(); 
        constructor.setAccessible(true); // Make the private constructor accessible
        SingletonClass instance2 = constructor.newInstance(); // Create a new instance
        System.out.println("Instance 2 hashcode: " + instance2.hashCode());
    }
}

```

### Serialization

During deserialization, a new instance of the class is created, breaking the singleton pattern:

&nbsp;

```java
public class SingletonBreakerDeseliazation {
    public static void main(String[] args) throws Exception {
    	//Converts an object into a byte stream for storage or transmission
        Singleton instance1 = Singleton.getInstance();
        //create a ObjectOutputStream to write to 
        ObjectOutputStream out = new ObjectOutputStream(new FileOutputStream("singleton.ser"));
        //write object to that file
        out.writeObject(instance1);
        out.close();
        
        //create new FileInputStreamfrom the same file
        //create ObjectInputStream  to read from
        ObjectInputStream in = new ObjectInputStream(new FileInputStream("singleton.ser"));
        Singleton instance2 = (Singleton) in.readObject();
        System.out.println(instance1 == instance2); // false
    }
}
```

&nbsp;

- **Serialization:** Converts an object into a byte stream for storage or transmission.
- **Deserialization:** Creates a new object from a serialized byte stream.
- **Singleton Vulnerability:** The ==default behavior of deserialization creates a new instance==, thus breaking the Singleton's single instance guarantee.

When we serialize and deserialize a singleton object, Java creates a new instance during deserialization, breaking the singleton pattern. This happens because deserialization doesn't use the constructor but instead reconstructs the object from bytes, bypassing the singleton logic.

meaning object with different hashCode is created when deserializing

### Strategies to Make Singleton Robust

1.  To prevent reflection:
    1.  **Ensure Single Instance**: The private constructor throws an exception if an instance already exists, preventing multiple instances via reflection.
        
    2.  **Thread-Safe Access**: The `INSTANCE` is created at class loading, ensuring thread safety without synchronization.
        

```java
public class RobustSingleton {
    private static final RobustSingleton INSTANCE = new RobustSingleton();
    
    private RobustSingleton() {
        if (INSTANCE != null) {
            throw new IllegalStateException("Singleton instance already exists!");
        }
    }
    
    public static RobustSingleton getInstance() {
        return INSTANCE;
    }
}
```

2\. To prevent serialization issues:

To prevent serialization from breaking your Singleton, you need to implement a special method called `readResolve()`. This method is called during deserialization and allows you to return the existing Singleton instance instead of creating a new one:

```java
public class SerializationRobustSingleton implements Serializable {
    private static final SerializationRobustSingleton INSTANCE = new SerializationRobustSingleton();
    
    private SerializationRobustSingleton() {}
    
    public static SerializationRobustSingleton getInstance() {
        return INSTANCE;
    }
    
    // This method is called during deserialization
    protected Object readResolve() {
        return INSTANCE;
    }
}
```
