### **Composition Over Inheritance:**

**Composition over Inheritance** is a design principle in object-oriented programming (OOP) that suggests using ==**composition** (combining simple objects to create more complex ones)== instead of **inheritance** (creating new classes by extending existing ones) for better flexibility and maintainability.

* * *

### **1\. What is Inheritance?**

Inheritance is when a **child class** (subclass) gets properties and methods from a **parent class** (superclass).  
Example:

```java
class Animal {
    void eat() { System.out.println("Eating..."); }
}

class Dog extends Animal { }  // Dog inherits from Animal

Dog dog = new Dog();
dog.eat();  // Output: "Eating..."
```

**Problem:** Inheritance creates a **tight coupling** between classes. If `Animal` changes, `Dog` may break.

* * *

### **2\. What is Composition?**

Composition means **building objects by combining other objects** (like Lego blocks). Instead of inheriting behavior, a class **contains** another class.  
Example:

```java
class Eater {
    void eat() { System.out.println("Eating..."); }
}

class Dog {
    private Eater eater = new Eater();  // Composition
    
    void eat() { eater.eat(); }  // Delegation
}

Dog dog = new Dog();
dog.eat();  // Output: "Eating..."
```

Here, `Dog` **has an `Eater`** instead of **being an `Animal`**.

* * *

### **3\. Why Composition is Better?**

| **Inheritance** | **Composition** |
| --- | --- |
| Tight coupling (changes in parent affect child) | Loose coupling (change one part without breaking others) |
| Hard to reuse (a `Dog` cannot easily reuse `Bird` behavior) | Easy reuse (just include the needed component) |
| Inflexible (cannot change behavior at runtime) | Flexible (can swap components at runtime) |

**Example Problem with Inheritance:**  
If you have `Bird` and `Dog` both extending `Animal`, but only `Bird` can `fly()`, you might end up with:

```java
class Animal {
    void eat() { ... }
    void fly() { ... }  // But Dog can't fly!
}
```

This leads to **unnecessary or incorrect methods** in subclasses.

**Solution with Composition:**

```java
class Flyer {
    void fly() { System.out.println("Flying..."); }
}

class Bird {
    private Flyer flyer = new Flyer();
    void fly() { flyer.fly(); }
}

class Dog {
    // No Flyer here! Only Birds can fly.
}
```

Now, `Dog` doesn’t have `fly()`, and `Bird` can still fly.

* * *

### **4\. When to Use Inheritance?**

Inheritance is still useful when:

- There’s a clear **"is-a"** relationship (e.g., `Dog` **is an** `Animal`).
- You need **polymorphism** (treating different subclasses the same way).

But **prefer composition** when:

- You need **flexibility** (e.g., changing behavior at runtime).
- You want **reusable, modular** code.

* * *

### **5\. Key Takeaways**

- **Composition = "Has-a"** (A `Car` **has an** `Engine`).
- **Inheritance = "Is-a"** (A `Dog` **is an** `Animal`).
- **Use composition** to avoid deep inheritance hierarchies and tight coupling.
- **Favor interfaces + composition** over big inheritance trees.

### **Example in Real Life:**

- **Inheritance:** A `Tesla` is a type of `ElectricCar`, which is a type of `Car`.
- **Composition:** A `Tesla` **has** a `Battery`, `Motor`, and `AutopilotSystem`.

Composition makes it easier to **swap** the `Battery` or upgrade `AutopilotSystem` without rewriting the whole class.

* * *

### **Final Thought**

> **"Inheritance breaks encapsulation, composition preserves it."**  
> – *Effective Java by Joshua Bloch*

By using composition, you keep your code **clean, flexible, and easy to maintain**! 🚀
