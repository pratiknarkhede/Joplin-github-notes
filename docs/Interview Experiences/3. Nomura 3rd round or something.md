## Equals, hashCode and Equals Contract

The equals contract in Java defines the relationship between the `equals()` and `hashCode()` methods.

The `equals()` method determines if two objects are equal based on their content, not just their memory references.

The `hashCode()` method generates an integer value (hash code) for an object, used in hash-based collections like HashMap and HashSet.

The equals contract states:

1.  If two objects are equal by `equals()`, they must have the same `hashCode()`
2.  If two objects have the same `hashCode()`, they are not necessarily equal by `equals()`
3.  If `equals()` is overridden, `hashCode()` must also be overridden to maintain this contract

```java
public class Person {
    private String name;
    private int age;
    
    // Constructor, getters, setters omitted
    
    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        
        Person person = (Person) obj;
        return age == person.age && 
               Objects.equals(name, person.name);
    }
    
    @Override
    public int hashCode() {
        return Objects.hash(name, age);
    }
}
```

## Default equals Implementation

If you don't override `equals()`, Java uses the default implementation from the Object class, which compares object references (memory addresses), not their content.

```java
// Default implementation in Object class
public boolean equals(Object obj) {
    return (this == obj);
}
```

This means two objects with identical content but different references will not be considered equal, which may cause unexpected behavior in collections.

## Abstract Class vs Interface

Abstract classes:

- Can have constructor methods
- Can have instance variables (fields)
- Can have concrete (implemented) methods
- Can have abstract methods that must be implemented by subclasses
- Support single inheritance only (a class can extend only one abstract class)
- Can have access modifiers for methods (public, private, protected)

Interfaces:

- Cannot have constructors
- Can only have constants (public static final variables)
- Can have default and static methods (since Java 8)
- Can have abstract methods (implicitly public)
- Support multiple inheritance (a class can implement multiple interfaces)
- All methods are implicitly public

## When to use Abstract Class vs Interface

Use an abstract class when:

- You want to share code among related classes
- Your classes need to have access to non-public methods and fields
- You want to have non-static, non-final fields
- You need to provide a common base implementation for related classes

Use an interface when:

- You want to define a contract that unrelated classes can implement
- You need multiple inheritance
- You want to specify behavior without dictating implementation
- You're designing for API flexibility where implementations might vary widely

## Need for Abstract Methods with Default Methods

Even though abstract classes can have default methods, abstract methods are still needed when:

1.  You want to enforce that subclasses provide their own implementation of certain behaviors
2.  The default implementation isn't appropriate or possible for all subclasses
3.  The behavior varies too much across subclasses to provide a meaningful default
4.  You need to create a template method pattern where some steps are fixed (default methods) and others are customizable (abstract methods)

## How Key Collision is Resolved in HashSet

HashSet uses HashMap internally. When a collision occurs:

1.  Java calculates the hash code of the object and determines its bucket location
2.  If the bucket already has an entry, Java uses a linked list (or a tree in Java 8+) to store multiple entries in the same bucket
3.  For each object in that bucket, Java compares the new object with existing ones using the `equals()` method
4.  If `equals()` returns true for any object, the new object is considered a duplicate
5.  If the linked list in a bucket grows too large (threshold is 8 in Java 8+), it's converted to a balanced tree for better performance

## Fail-fast and Fail-safe Iterators

Fail-fast iterators:

- Throw ConcurrentModificationException if the collection is modified while iterating
- Work with collections like ArrayList, HashMap, Vector
- Keep track of modifications using a modification counter
- Are typically faster and use less memory

```java
ArrayList<String> list = new ArrayList<>();
list.add("Apple");
list.add("Banana");

// Fail-fast iterator
Iterator<String> iterator = list.iterator();
while (iterator.hasNext()) {
    String fruit = iterator.next();
    // This would throw ConcurrentModificationException:
    // list.remove("Apple");
    
    // This is the correct way to remove:
    // iterator.remove();
}
```

Fail-safe iterators:

- Work on a copy of the collection, so they don't throw exceptions if the original collection is modified
- Are found in concurrent collections like CopyOnWriteArrayList, ConcurrentHashMap
- Use more memory due to the collection copy
- May not reflect the latest state of the collection

## ArrayList vs LinkedList

ArrayList:

- Backed by a dynamic array
- Fast random access (O(1)) using index
- Slow insertion/deletion in the middle (O(n)) due to element shifting
- Better for scenarios with frequent access and rare modifications
- More memory efficient for storage

LinkedList:

- Implemented as a doubly-linked list
- Slow random access (O(n))
- Fast insertion/deletion (O(1)) when you have a position reference
- Better for scenarios with frequent modifications
- Uses more memory due to node references

## HashTable Implementation and Thread Safety

HashTable is implemented as an array of buckets, each containing key-value entries. It's thread-safe because:

1.  All methods are synchronized at the object level
2.  Only one thread can access the HashTable at a time (coarse-grained locking)
3.  This synchronization happens on the entire table, causing performance bottlenecks

```java
// Simplified implementation of a put method in Hashtable
public synchronized V put(K key, V value) {
    // Implementation details
    return oldValue;
}
```

This makes HashTable thread-safe but less efficient than HashMap for single-threaded applications.

## ConcurrentHashMap Read/Write Operations

In ConcurrentHashMap:

1.  The map is divided into segments (16 by default in older versions)
2.  Each segment has its own lock
3.  Multiple threads can read from different segments simultaneously
4.  When one thread is writing to a segment, other threads can still read from or write to different segments
5.  Reads don't require a lock (non-blocking reads)
6.  Writes only lock the affected segment, not the entire map
7.  In Java 8+, it uses a more efficient approach with node-level locking

This gives better concurrency than Hashtable while maintaining thread safety.

## Design Principles

DRY (Don't Repeat Yourself):

- Avoid code duplication
- Extract common functionality into reusable methods or classes
- Helps maintenance since changes only need to be made in one place

SOLID principles:

- Single Responsibility: A class should have only one reason to change
- Open/Closed: Open for extension, closed for modification
- Liskov Substitution: Subtypes must be substitutable for their base types
- Interface Segregation: Many client-specific interfaces are better than one general-purpose interface
- Dependency Inversion: Depend on abstractions, not on concrete implementations

KISS (Keep It Simple, Stupid):

- Avoid unnecessary complexity
- Simple solutions are easier to understand, maintain, and debug
- Focus on solving the current problem, not potential future problems

## Java LinkedList Implementation

Java's LinkedList is implemented as a doubly-linked list:

- Each node has references to both the previous and next nodes
- This allows traversal in both directions
- It implements both List and Deque interfaces
- Provides efficient operations at both ends of the list

## Lock Priority

In Java's concurrency model:

- There is no built-in concept of lock priorities
- Thread priorities don't directly affect lock acquisition
- Locks are generally acquired on a first-come, first-served basis
- For custom priority behavior, you would need to implement priority-based synchronization using explicit constructs like ReentrantLock with custom fairness policies or priority queues

Instead of relying on lock priorities, it's better to design your concurrency approach based on your specific requirements, possibly using higher-level concurrency utilities from the java.util.concurrent package.
