## What is the Builder Pattern?

The Builder pattern is a creational ==design pattern that lets you construct complex objects step by step==. It allows you to produce different types and representations of an object using the same construction code.

&nbsp;

## When to Use It?

Use the Builder pattern when:

1.  You need to create an object with many possible configuration options.
2.  You want to create immutable objects.
3.  You need to construct objects that contain multiple parts.
4.  You want to separate the construction of a complex object from its representation.

&nbsp;

## Use in Spring Boot Development

&nbsp;

### 1\. Lombok @Builder

Lombok's `@Builder` annotation is a popular way to implement the Builder pattern in Spring Boot applications:

```java
    import lombok.Builder;
import lombok.Data;

@Data
@Builder
public class User {
    private String username;
    private String email;
    private String firstName;
    private String lastName;
    private int age;
}

// Usage
User user = User.builder()
    .username("johndoe")
    .email("john@example.com")
    .firstName("John")
    .lastName("Doe")
    .age(30)
    .build();
```

&nbsp;

`@Builder`: it automatically generates the code necessary to implement the Builder pattern for our class

- Create a nested static class named `UserBuilder` within your `User` class.
- Generate a `builder()` method in your `User` class that returns a new instance of `UserBuilder`.
- Create setter-like methods (with the same names as your fields) in the `UserBuilder` class. These methods return the builder instance itself, enabling method chaining.
- Generate a `build()` method in the `UserBuilder` class that constructs and returns a `User` object using the values set in the builder.

### 2\. Spring's WebClient

Spring WebFlux's `WebClient` uses the Builder pattern for configuration:

```java
WebClient webClient = WebClient.builder()
    .baseUrl("https://api.example.com")
    .defaultHeader(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
    .defaultUriVariables(Collections.singletonMap("url", "https://api.example.com"))
    .build();
```

`WebClient.Builder `    is static nested inner class inside `WebClient`

- **Obtain Builder:** `WebClient.builder()` gives you an instance of the `WebClient.Builder` class, which is the starting point for your configuration.
    
- **Chain Configuration:** You chain method calls on the builder to set different properties. Each method (like `baseUrl`, `defaultHeader`, `defaultUriVariables`) returns the builder instance, allowing you to fluently add more configurations.
    
- **Build the Client:** Finally, calling `build()` creates the fully configured `WebClient` instance that you can use to make HTTP requests.

### 3\. TestRestTemplate Builder

In Spring Boot testing, `TestRestTemplate.Builder` is used to configure the template:

```java
@Autowired
private TestRestTemplate.Builder builder;

@Test
public void testSomething() {
    TestRestTemplate template = builder
        .basicAuthentication("user", "password")
        .build();
    
    // Use the template for testing
}
```

> we use `TestRestTemplate `   while doing intergration testing over `restTemplate` since it is designed to work seamlessly with the Spring Boot testing framework. It automatically configures itself to interact with your application's embedded server or a running instance if you've specified a port.
> 
> `TestRestTemplate` can be easily mocked or stubbed for unit tests that don't require a full integration test environment.
> 
> When using `@SpringBootTest`, the `TestRestTemplate` is auto-configured, and you can inject it directly into your test class.
> 
> Easier Authentication: Built-in support for basic auth and extensibility for other mechanisms.

&nbsp;

**code flow:**

- **Builder Injection:**
    
    - Spring Boot auto-configures a `TestRestTemplate.Builder` bean.
    - You inject this builder into your test class using `@Autowired`.
- **Configuration Chaining:**
    
    - You call methods on the builder (e.g., `.basicAuthentication()`) to configure the `TestRestTemplate`.
    - Each configuration method returns the builder instance, allowing you to chain calls together.
- **Building the Template:**
    
    - The `.build()` method creates a `TestRestTemplate` instance with the specified configuration. This instance is ready to make HTTP requests.

### 4\. Custom Configuration Builder

You might create a builder for complex configuration objects:

```java
@Configuration
public class DatabaseConfig {
    
    @Bean
    public DataSource dataSource() {
        return new DataSourceBuilder()
            .url("jdbc:mysql://localhost:3306/mydb")
            .username("user")
            .password("password")
            .driverClassName("com.mysql.jdbc.Driver")
            .build();
    }
}
```

### 5\. Building Complex Domain Objects

For complex domain objects, a custom builder can be very useful:

```java
public class Order {
    private final String id;
    private final Customer customer;
    private final List<OrderItem> items;
    private final BigDecimal total;
    private final LocalDateTime orderDate;
    
    private Order(Builder builder) {
        this.id = builder.id;
        this.customer = builder.customer;
        this.items = builder.items;
        this.total = builder.total;
        this.orderDate = builder.orderDate;
    }
    
    public static class Builder {
        private String id;
        private Customer customer;
        private List<OrderItem> items = new ArrayList<>();
        private BigDecimal total = BigDecimal.ZERO;
        private LocalDateTime orderDate;
        
        public Builder id(String id) {
            this.id = id;
            return this;
        }
        
        public Builder customer(Customer customer) {
            this.customer = customer;
            return this;
        }
        
        public Builder addItem(OrderItem item) {
            this.items.add(item);
            this.total = this.total.add(item.getPrice());
            return this;
        }
        
        public Builder orderDate(LocalDateTime orderDate) {
            this.orderDate = orderDate;
            return this;
        }
        
        public Order build() {
            return new Order(this);
        }
    }
}

// Usage
Order order = new Order.Builder()
    .id("12345")
    .customer(customer)
    .addItem(new OrderItem("Item1", new BigDecimal("10.00")))
    .addItem(new OrderItem("Item2", new BigDecimal("15.00")))
    .orderDate(LocalDateTime.now())
    .build();
```

### 6\. Spring Security

Spring Security uses the Builder pattern for configuring security rules:

&nbsp;

&nbsp;

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/", "/home").permitAll()
                .anyRequest().authenticated()
                .and()
            .formLogin()
                .loginPage("/login")
                .permitAll()
                .and()
            .logout()
                .permitAll();
    }
}
```

&nbsp;

- **Builder Object:**
    
    - The `HttpSecurity` object you receive in the `configure` method acts as the builder. It provides methods like `authorizeRequests()`, `formLogin()`, `logout()`, and more.
- **Chaining Methods:**
    
    - Each method on the `HttpSecurity` builder returns the builder instance itself. This allows you to chain calls together in a fluent manner. For example: `http.authorizeRequests()...and().formLogin()...and().logout()...`
- **Build (Implicit):**
    
    - In this specific case, there's no explicit `build()` method like in some other builder implementations. The Spring Security configuration is built implicitly when the `configure()` method completes.

* * *

> **Fluent Interface**
> 
> A fluent interface is a design pattern where methods return the object they are called on (or a similar object) to allow for method chaining. This makes the code more concise and readable.
