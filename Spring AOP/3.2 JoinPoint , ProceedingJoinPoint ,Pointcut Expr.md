A JoinPoint is a point during the execution of a program, such as the execution of a method or the handling of an exception.

**In Spring AOP, a JoinPoint always represents a method execution.**

```java
@Before("execution(* com.example.service.*.*(..))")
public void beforeAdvice(JoinPoint joinPoint) {
    // ...
}
```

The JoinPoint object provides useful information about the method being executed:

- `joinPoint.getSignature()`: Returns a Signature object, which contains information about the method being called.
- `joinPoint.getSignature().getName()`: Gets the name of the method being called.
- `joinPoint.getArgs()`: Returns an array of the arguments passed to the method.
- `joinPoint.getTarget()`: Returns the target object being advised.

Aspects take JoinPoint (or ProceedingJoinPoint for @Around advice) as an argument to provide context about the method being advised. This allows the aspect to access information about the method, its arguments, the object it's being called on, etc. 

&nbsp;

**ProceedingJoinPoint:**

ProceedingJoinPoint is a subinterface of JoinPoint. It's used specifically with @Around advice and adds the ability to control whether the advised method is actually executed or not.

```java
@Around("execution(* com.example.service.*.*(..))")
public Object aroundAdvice(ProceedingJoinPoint joinPoint) throws Throwable {
    // Before advice
    Object result = joinPoint.proceed(); // Execute the advised method
    // After advice
    return result;
}
```

&nbsp;The key difference is the `proceed()` method, which allows you to execute the advised method.

**Pointcut Expressions**

The expression `execution(* com.example.service.*.*(..))` is a pointcut expression. Let's break it down:

- `execution`: This designator means the join point is the execution of a method.
- `*`: The first asterisk is a wildcard representing any return type.
- `com.example.service`: This is the package name.
- `*`: The second asterisk is a wildcard representing any class in the specified package.
- `*`: The third asterisk is a wildcard representing any method name.
- `(..)`: The double dots represent any number of parameters (zero or more).

So, this expression matches the execution of any method in any class in the com.example.service package, regardless of its return type or parameters.

**joinPoint.proceed()**

Imagine you have a regular method in your code, like this:

```java
public void doSomething() {
    System.out.println("Doing something");
}
```

&nbsp;Now, when you use @Around advice, it's like wrapping this method in another method. The `joinPoint.proceed()` call is essentially saying "now run the original method". Here's a visualization

```java
@Around("execution(* doSomething())")
public Object aroundAdvice(ProceedingJoinPoint joinPoint) throws Throwable {
    System.out.println("Before doing something");
    
    Object result = joinPoint.proceed(); // This runs the original doSomething() method
    
    System.out.println("After doing something");
    return result;
}
```

So, when you call `doSomething()`, it actually runs:

1.  "Before doing something"
2.  The original "Doing something"
3.  "After doing something"

now this is for single method called doSomething(), we can set this to match every method in a package  
it will be like wrapping that method inside other method where when we decide what code needs to be executed before and after the actual method execution

**ProceedingJoinPoint vs JoinPoint** 

Imagine you're a security guard (the aspect) at a party (your program).

With a regular JoinPoint (used in @Before, @After, etc.), you're like a guard who can see who's coming in and out, but you can't stop them. You can only watch and take notes.

With a ProceedingJoinPoint (used in @Around), you're like a guard who can actually stop people, check their IDs, maybe even turn them away. The `proceed()` method is like you saying "Okay, you can go in now" to a guest.

&nbsp;In code terms:

- With JoinPoint, you can see information about the method being called, but you can't control its execution.
- With ProceedingJoinPoint, you can control whether the method runs, when it runs, and even change what it returns.

&nbsp;**Example:**

Following example explains how ProceedingJoint has control over your method execution  
<br/>

- We have a `BankAccount` class with a `withdraw` method.
- The `SecurityAspect` uses @Around advice to "wrap" the withdraw method.
- In the `secureWithdrawal` method:
    - We check the withdrawal amount.
    - If it's over $1000, we don't allow the withdrawal (we don't call `proceed()`).
    - If it's $1000 or less, we allow the withdrawal by calling `proceed()`.
- The `proceed()` call is what actually runs the original `withdraw` method.

```java
import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.Around;
import org.aspectj.lang.annotation.Aspect;
import org.springframework.stereotype.Component;

@Aspect
@Component
public class SecurityAspect {

    @Around("execution(* com.example.BankAccount.withdraw(..))")
    public Object secureWithdrawal(ProceedingJoinPoint joinPoint) throws Throwable {
        Object[] args = joinPoint.getArgs();
        double amount = (Double) args[0];

        if (amount > 1000) {
            System.out.println("Withdrawal amount exceeds limit. Cancelling transaction.");
            return false;  // Don't proceed with the withdrawal
        }

        System.out.println("Withdrawal amount within limit. Proceeding with transaction.");
        return joinPoint.proceed();  // Proceed with the original withdrawal method
    }
}

public class BankAccount {
    public boolean withdraw(double amount) {
        System.out.println("Withdrawing $" + amount);
        // Actual withdrawal logic here
        return true;
    }
}
```

&nbsp;

So, if someone tries to withdraw $500, the code will:

1.  Print "Withdrawal amount within limit. Proceeding with transaction."
2.  Then run the original `withdraw` method, printing "Withdrawing $500"

But if someone tries to withdraw $1500, it will:

1.  Print "Withdrawal amount exceeds limit. Cancelling transaction."
2.  Return false, without ever running the original `withdraw` method
