**CAP Theorem** states that in a distributed system, you can only achieve **two** of the following three  desirable properties at any given time:

1.  **Consistency** (C): All nodes see the same data at the same time.
2.  **Availability** (A): The system continues to function even if some nodes fail.
3.  **Partition Tolerance** (P): The ability of system to continue to work despite network partitions

&nbsp;

&nbsp;

You must **choose** between:

- **CA (Consistency + Availability)**: No partition tolerance.
- **CP (Consistency + Partition Tolerance)**: Some nodes might be unavailable.
- **AP (Availability + Partition Tolerance)**: Data may become inconsistent.

&nbsp;

### What is CAP Theorem?

The **CAP theorem** (also called Brewer's theorem) describes the limitations a distributed data store faces regarding Consistency, Availability, and Partition Tolerance. It was introduced by Eric Brewer.

&nbsp;

### Meaning of CAP Properties:

Assuming we have different nodes having data replicated across them ,

1.  **Consistency (C)**: Every read reflects the latest write. All nodes return the same data. For example, in a banking system, if you withdraw money, all nodes should reflect the updated balance immediately.  
    i.e  If we do write a=5 on node 1 it should be synced across other nodes so that if we read data from other node it should have value 5  
    <br/>
2.  **Availability (A)**: Every request receives a response (success/failure), even when some nodes are down.  
    At least one node should respond (not necessarily process the request) to the request   
    <br/>
3.  **Partition Tolerance (P)**: **The system continues to operate despite network failures that split communication between nodes.** For instance, in a microservices setup, even if some services cannot communicate with each other, the system still works.  
    i.e suppose there 2 nodes in DS , even if the communication link between these two fails , they both or one still responding to the system  
    <br/><br/>

### Why all three (C, A, P) are not possible?

Suppose we want to have all 2 nodes having data replicated across them, and despite communication link failure between these nodes A and B , they are still serving client request  
<br/>so here we can see that **P** is satisfied, and also **A** is satisfied, but since there is no communication link between A and B  
Consistency cannot be satisfied ,since we writes cannot be synced or reflected if it happenes on either A and B and hence they will give inconsistent response

When a **network partition** occurs (which will eventually happen in any distributed system):

- You **can't** keep data perfectly **consistent** across all nodes **and** guarantee that every request is served immediately (available).
- You have to choose one: **Consistency or Availability**.

&nbsp;

### Why Partition Tolerance is Essential:

Network partitions **will happen** in a distributed system, especially when nodes are geographically distributed. If your system can’t tolerate partitions, it becomes unusable whenever there's a network failure.

==Partition Tolerance ensures that your system **continues to function** even when some nodes can’t talk to each other. However, during this time, you have to choose between ensuring the **same data** (Consistency) or ensuring the system **keeps running** (Availability).==

- **PA (Partition Tolerance + Availability)**: You choose to keep the system up and running (Availability) even during a network partition, but the data may become inconsistent between nodes. This is common in many distributed NoSQL databases (e.g., Cassandra, DynamoDB), where **eventual consistency** is acceptable.
    
- **PC (Partition Tolerance + Consistency)**: You ensure that the data remains consistent across all nodes, even if some parts of the system become unavailable during a partition. Systems like HBase prioritize strong consistency but may sacrifice availability if a partition occurs.
    
- **AC (Availability + Consistency)**: **This combination is desirable, but it cannot tolerate network partitions.** It can be used in **centralized systems** or systems that don’t need to worry about partitions, such as:
    
    - **Single-node databases** (e.g., MySQL on a single server).
    - **Non-distributed systems**: Here, you don’t need partition tolerance because all the data is on one machine. The system can be consistent and available as long as that single node is up.

**AC** is possible in **non-distributed systems** or systems with **no network partitions**, but **not in distributed systems with partitions**.

&nbsp;

However, once you introduce the possibility of **partitions**, maintaining both **A** and **C** is impossible, and you have to pick between them.
