### What is a Database Index?

A database index is a ==data structure that improves the speed of data retrieval operations on a database table at the cost of additional writes and storage space.== Indexes are used to quickly locate data without having to search every row in a table when a query is processed.

&nbsp;

### Physical Implementation

At the physical level, an index is stored as a separate file on disk that contains:

- **Keys**: Sorted values from the indexed column(s)
- **Pointers**: References to the physical location (block/page addresses) where the actual data records are stored

this separate file is in sorted order     

&nbsp;

&nbsp;

## Data Structures Used for Indexing

Most databases use one of these structures for indexes:

1.  **B-tree/B+ tree**: The most common index type
2.  **Hash indexes**: For exact match queries
    1.  Hash indexes use a hash function to map keys directly to bucket locations
3.  **Bitmap indexes**: For low-cardinality data
    1.  **R-tree**: For spatial data

&nbsp;

### B-trees

- Self-balancing tree structure
- Maintains sorted data for efficient searches, insertions, and deletions
- All leaf nodes are at the same depth (balanced)
- Optimized for systems that read and write large blocks of data

&nbsp;

Imagine a table of employees:

```
  employee_id  | name          | department  | salary
    -----------+---------------+-------------+--------
        1          | John Smith    | Engineering | 85000
        2          | Sarah Johnson | Marketing   | 75000
        3          | Mike Williams | Finance     | 92000
        4          | Lisa Brown    | Engineering | 88000
        5          | David Miller  | Sales       | 78000
```

&nbsp;

Without an index, searching for "salary > 80000" requires scanning every row.

With a B-tree index on the salary column, the database creates a structure like:

```
               [75000, 85000]
              /              \
     [75000, 78000]        [85000, 92000]
     /         \           /          \
  [75000]    [78000]    [85000, 88000]  [92000]
    |          |        /        \        |
   ID:2       ID:5    ID:1      ID:4     ID:3
```

&nbsp;

&nbsp;

### Clustered Indexes

==A clustered index determines the physical order of data in a table(done on original physically stored data):==

- Only one clustered index per table (because data can only be physically organized in one way)
- Usually implemented on the primary key
- Actual data rows are stored in the leaf nodes of the index
- When created, the table's rows are reordered to match the index
- No separate pointer is needed because the data itself is stored in order

&nbsp;

**Physical Implementation:**

- The table data is reorganized physically on disk to match the clustered index order
- No separate pointer is needed because the data itself is stored in order

&nbsp;

* * *

### Non-Clustered Indexes

Non-clustered indexes maintain a separate structure from the data:

- Does **not alter the physical order** of data
    
- You can have **multiple** non-clustered indexes on a table
    
- Contains a **sorted reference** to the data (either a row pointer or the clustered index key)
    
- The leaf nodes contain pointers to the actual data rather than the data itself
    
- Faster for insert/update operations than clustered indexes
    
- Useful for columns frequently used in WHERE clauses that aren't the primary key
    

&nbsp;

**Physical Implementation:**

- ==Maintained as a separate file with keys and pointers==
- Each entry in the leaf node points to the data page containing the full record

## How Indexes Work in Practice

### 4.1 Query Processing with Indexes

When a database processes a query with WHERE conditions:

1.  Query optimizer checks available indexes
2.  Evaluates cost of using indexes vs. table scan
3.  For chosen index, traverses the index structure from root to leaves
4.  Follows pointers to retrieve actual data records
5.  Assembles result set

**Example:**

```sql
-- With index on salary
SELECT * FROM employees WHERE salary > 80000;
```

Execution steps:

1.  Start at root node of salary index
2.  Traverse to find first value >= 80000
3.  Scan leaf nodes from that point, following pointers to data
4.  Retrieve only records matching the condition\\

&nbsp;

* * *

Database data is stored in fixed-size pages (typically 4KB, 8KB, or 16KB):

- Index helps minimize the number of pages that must be read
- Without an index: sequential scan reads all pages
- With an index: reads only index pages + relevant data pages

* * *

&nbsp;

### Index Maintenance

Indexes are dynamically maintained during DML operations:

- INSERT: New key added to appropriate location in index
- UPDATE: If indexed column changes, index entry is relocated
- DELETE: Index entry is marked as deleted or removed

This maintenance adds overhead to write operations.
