### 1.1 What is a Database Transaction?

A database transaction is a s==equence of one or more SQL operations (read/write) that are executed as a single logical unit of work.== The operations either all succeed (commit) or all fail (rollback), maintaining the database in a consistent state. This concept is critical for maintaining data integrity in applications.

### 1.2 Core Properties: ACID

Database transactions are characterized by four fundamental properties, commonly referred to as ACID:

1.  **Atomicity**: Transactions are all-or-nothing operations. Either all the operations within a transaction succeed, or none of them take effect.
2.  **Consistency**: A transaction brings the database from one valid state to another valid state, maintaining all predefined rules including constraints, cascades, triggers, etc.
3.  **Isolation**: Concurrent execution of transactions results in a state that would be obtained if they were executed sequentially. Each transaction must operate as if it is the only transaction running on the system.
4.  **Durability**: Once a transaction is committed, its effects are permanently stored in the database, surviving system failures, crashes, or power outages.

* * *

### 2.1 Transaction States

A transaction passes through several states during its lifecycle:

1.  **Active**: Initial state; the transaction stays in this state while executing its statements
2.  **Partially Committed**: After the final statement has been executed
3.  **Failed**: When normal execution can no longer proceed
4.  **Aborted**: After the transaction has been rolled back and the database has been restored to its state prior to the transaction
5.  **Committed**: After successful completion

## Transaction Isolation Levels

### 3.1 Concurrency Problems

When multiple transactions execute concurrently, several problems can occur:

1.  **Dirty Read**: A transaction reads data that has been modified by another transaction that has not yet been committed.
2.  **Non-repeatable Read**: A transaction re-reads data it has previously read and finds that another committed transaction has modified or deleted the data.
3.  **Phantom Read**: A transaction re-executes a query returning a set of rows that satisfy a condition and finds that another committed transaction has inserted additional rows that satisfy the condition.
4.  **Lost Update**: Two transactions select the same row, then first transaction updates the row based on the originally selected values, then the second transaction updates the same row, overwriting the first transaction's update.

&nbsp;

### Isolation Levels

| Level | Description | Problem It Solves |
| --- | --- | --- |
| **Read Uncommitted** | Dirty reads allowed (see uncommitted changes). | Avoid if data accuracy is critical. |
| **Read Committed** | Only see committed data. | Prevents dirty reads. |
| **Repeatable Read** | Data remains consistent within the transaction. | Prevents non-repeatable reads. |
| **Serializable** | Highest isolation (no concurrency). | Prevents phantom read |

&nbsp;

MySQL has default isolation level as REPEATABLE READ

## 4\. Transaction Implementation Mechanisms

### 4.1 Locking Mechanisms

Databases use various locking strategies to implement transaction isolation:

1.  **Shared (S) Locks**: Used for read operations. Multiple transactions can acquire shared locks on the same data simultaneously.
2.  **Exclusive (X) Locks**: Used for write operations. Only one transaction can hold an exclusive lock on a piece of data at any given time.
3.  **Update (U) Locks**: Used in some systems as an intermediary between shared and exclusive locks.
4.  **Intent Locks**: Indicate that a transaction intends to acquire locks at a finer granularity level.

&nbsp;

&nbsp;

### 4.2 Lock Granularity

Locks can be applied at different levels:

1.  **Row-level Locks**: Lock individual rows, allowing high concurrency but with higher overhead.
2.  **Page-level Locks**: Lock pages (blocks) of data, balancing concurrency and overhead.
3.  **Table-level Locks**: Lock entire tables, reducing overhead but limiting concurrency.
4.  **Database-level Locks**: Lock the entire database, used for global operations.
