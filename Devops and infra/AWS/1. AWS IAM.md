### **What is AWS IAM?**

**IAM = Identity and Access Management**  
Imagine a bouncer at a nightclub who checks IDs and decides who gets in. IAM is AWS's "bouncer" that:

- ==Controls **WHO** (users/roles) can do **WHAT** (actions) on **WHICH** AWS resources (e.g., EC2, S3).==
    
- Manages permissions through **policies** (JSON rules).
    

&nbsp;

&nbsp;

## <span style="color: #ffffff;">Core Idea 1 – Resources</span>

**Resources** are the entities that you create in AWS. This can include thing’s like an S3 bucket, an S3 Object, or a DynamoDB tables. All resources in AWS are represented by something called an **ARN** or **A**mazon **R**esource **N**ame.   `arn:aws:s3:::test-website-awssimplified`<span style="color: #ffffff;">.</span>

&nbsp;

## <span style="color: #ffffff;">Core Idea 2 – Actions</span>

**Actions** are the operations that users attempt to perform on resources.

For example, if I created a lambda function, I may want to update its configuration. In order to do that, I need to use an API called `lambda:UpdateFunctionConfiguration`. In order to perform this action, I need to have permissions to call this API. But how do I get permissions?

The answer lies in the **Policy** …

&nbsp;

## <span style="color: #ffffff;">Core Idea 3 – Policy / Policy Document</span>

The heart of AWS IAM is the **Policy**. IAM policies refer specific JSON documents that define user permissions to access a resource. In other words, authorization to perform an **Action** is determined by whether or not the user has the appropriate **Policy Document** with associated permissions.

&nbsp;

* * *

&nbsp;

#### 1\. **Users**

- A **user** represents an individual or service that interacts with AWS (e.g., a developer, an EC2 instance, or your Java app).
- Each user has **credentials** (like a username/password or **access keys**) to authenticate with AWS.
- Example: A Java app using the AWS SDK needs access keys (like `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY`) to interact with AWS services.

#### 2\. **Groups**

- A **group** is a collection of users. Instead of assigning permissions to individual users, you assign them to a group.
- Example: A group called `Developers` might have permissions to access S3 and DynamoDB, while a group called `Admins` has full access.

#### 3\. **Roles**

- ==A **role** is an identity with temporary credentials== that you can assign to a trusted entity (like an EC2 instance, Lambda function, or another AWS service).
- Example: Your Java app running on an EC2 instance can assume a role to access an S3 bucket without needing long-term access keys. This is more secure!

#### 4\. **Policies**

- **Policies** are ==JSON documents that define permissions== (e.g., "allow read access to S3 bucket X").
- Policies are attached to users, groups, or roles.
- <span style="color: #fafafc;">Example Policy (allow read-only access to an S3 bucket):</span>

&nbsp;

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

&nbsp;<span style="color: #ffffff;">there can be multiple statements within a single policy document that define access to different resources</span>

<span style="color: #ffffff;"><span style="color: #ffffff;">Explicit Denies always take priority over Allows, so if there is ever a conflict, Deny will win.</span></span>

&nbsp;

#### 5\. **Permissions**

- **Permissions** determine what actions (e.g., `s3:PutObject`, `dynamodb:Query`) are allowed or denied on specific resources.

&nbsp;

#### **Security Token Service (STS)**

- A service that generates **temporary credentials** (e.g., for assuming roles or federated users).  
    Example: Your Java app can use STS to assume a role and get short-lived credentials to access another AWS account’s resources.

&nbsp;

**What is the principle of least privilege, and how do you apply it in IAM?**  
*Expected Answer* :  
Grant only the permissions necessary for a task. For example, if a Java app only needs to read from S3, don’t grant write/delete permissions.

&nbsp;

**Why are IAM roles preferred over access keys for EC2 instances?**  
*Expected Answer* :  
Roles provide temporary credentials, reducing the risk of long-term credential leaks. Access keys are static and harder to rotate.

**How do you audit IAM permissions for a Java application?**  
*Expected Answer* :  
Use AWS IAM Access Analyzer or AWS CloudTrail to review permissions and detect unused roles/policies.

&nbsp;

#### **What’s the difference between IAM roles and users?**

- **Users**: Long-term credentials (passwords/keys) for people/apps.
    
- **Roles**: Temporary credentials for AWS services (EC2, Lambda) or federated users (SAML).
