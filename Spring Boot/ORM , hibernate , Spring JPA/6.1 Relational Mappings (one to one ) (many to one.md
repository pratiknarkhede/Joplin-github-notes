# One-to-One Unidirectional Relationship

## Basic Concept

A one-to-one unidirectional relationship is where one entity references another, but not vice versa. Think of it like a person having a passport - the person knows about their passport, but the passport doesn't know about its owner (in the database relationship sense).

example using Employee and EmployeeDetail entities:

&nbsp;

```java
@Entity
@Table(name = "employee")
public class Employee {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String name;
    
    @OneToOne(cascade = CascadeType.ALL)
    @JoinColumn(name = "detail_id")
    private EmployeeDetail employeeDetail;
    
    // getters and setters
}
```

observe we only mentioned the reference table name here and by default it will treat primary key of that table as foreign key in the current table

```java
@Entity
@Table(name = "employee_detail")
public class EmployeeDetail {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String address;
    private String phoneNumber;
    
    // getters and setters
}
```

&nbsp;

In this example, the Employee has a reference to EmployeeDetail, but EmployeeDetail doesn't know about the Employee. This is a unidirectional relationship.

The SQL that Hibernate generates for the employee table will look something like:

```sql
CREATE TABLE employee (
    id BIGINT NOT NULL AUTO_INCREMENT,
    name VARCHAR(255),
    detail_id BIGINT,
    PRIMARY KEY (id),
    FOREIGN KEY (detail_id) REFERENCES employee_detail(id)
);
```

&nbsp;If we want to use our own custom name for the join column, we can specify it in the `@JoinColumn` annotation:

&nbsp;

```java
@OneToOne(cascade = CascadeType.ALL)
@JoinColumn(name = "employee_detail_fk", referencedColumnName = "id")
private EmployeeDetail employeeDetail;
```

This would create a foreign key column named `employee_detail_fk` instead of the default `detail_id`.

## One-to-One with Composite Key

When dealing with composite keys, we need to use a different approach. Let's create an example with a composite key:

<span style="color: #d1d5db;">A composite key is defined using</span> `@Embeddable` <span style="color: #d1d5db;">and embedded into the entity using</span> `@EmbeddedId`

```java
@Embeddable
public class EmployeeDetailId implements Serializable {
    private String departmentCode;
    private String employeeCode;
    
    // constructors, equals, hashCode
}
```

&nbsp;

```java
@Entity
@Table(name = "employee_detail")
public class EmployeeDetail {
    @EmbeddedId
    private EmployeeDetailId id;
    
    private String address;
    private String phoneNumber;
    
    // getters and setters
}
```

&nbsp;In below case, we use `@JoinColumns` (plural) to specify multiple join columns, one for each component of the composite key.

i.e. <span style="color: #d1d5db;">The</span> `@OneToOne` <span style="color: #d1d5db;">relationship is established using</span> `@JoinColumns` <span style="color: #d1d5db;">to map multiple columns in the owning entity (</span>`Employee`<span style="color: #d1d5db;">) to the composite key of the target entity (</span>`EmployeeDetail`<span style="color: #d1d5db;">).</span>

```java
@Entity
@Table(name = "employee")
public class Employee {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String name;
    
    @OneToOne(cascade = CascadeType.ALL)
    @JoinColumns({
        @JoinColumn(name = "department_code", referencedColumnName = "departmentCode"),
        @JoinColumn(name = "employee_code", referencedColumnName = "employeeCode")
    })
    private EmployeeDetail employeeDetail;
    
    // getters and setters
}
```

## Cascade Types in Detail

Cascade operations determine what happens to related entities when you perform an operation on the owning entity. Let's explore each type:

### CascadeType.PERSIST

This cascades the persist (save) operation from the owning entity to the related entity.

```java
@OneToOne(cascade = CascadeType.PERSIST)
private EmployeeDetail employeeDetail;
```

With this setting, if you save an Employee, the associated EmployeeDetail will also be saved:

```java
EmployeeDetail detail = new EmployeeDetail();
detail.setAddress("123 Main St");
detail.setPhoneNumber("555-1234");

Employee employee = new Employee();
employee.setName("John Doe");
employee.setEmployeeDetail(detail);

// Only need to save the employee, the detail will be saved too
employeeRepository.save(employee);
```

&nbsp;

### CascadeType.MERGE

This cascades the merge operation, which is useful when you want to update both the owning and related entities.

```java
@OneToOne(cascade = CascadeType.MERGE)
private EmployeeDetail employeeDetail;

```

&nbsp;

```java
Employee employee = employeeRepository.findById(1L).get();
employee.setName("Jane Doe");
employee.getEmployeeDetail().setAddress("456 Oak St");

// Updates both employee and employeeDetail
employeeRepository.save(employee);
```

&nbsp;

### CascadeType.REMOVE

This cascades the remove operation, deleting the related entity when the owning entity is deleted.

```java
@OneToOne(cascade = CascadeType.REMOVE)
private EmployeeDetail employeeDetail;
```

&nbsp;

```java
Employee employee = employeeRepository.findById(1L).get();
employeeRepository.delete(employee); // This will also delete the associated EmployeeDetail
```

&nbsp;

### CascadeType.REFRESH

This cascades the refresh operation, which reloads the state of both the owning and related entities from the database.

meaning it will skip the first level cache check and will directly fetch from db

&nbsp;

### CascadeType.ALL

This cascades all operations (PERSIST, MERGE, REMOVE, REFRESH, DETACH).

## Fetch Types: Eager vs. Lazy Loading

While cascade types control what happens with write operations, **fetch types** control how entities are loaded from the database.(concerns loading of child entities if parent owning entity is loaded)

### Eager Loading

Eager loading loads the related entity immediately when the owning entity is loaded.

```java
@OneToOne(fetch = FetchType.EAGER)
private EmployeeDetail employeeDetail;
```

This is the **default** for `@OneToOne` and `@ManyToOne` relationships.

Example:

```java
Employee employee = employeeRepository.findById(1L).get();
// employeeDetail is already loaded
String address = employee.getEmployeeDetail().getAddress();
```

&nbsp;

* * *

### Lazy Loading

Lazy loading defers loading the related entity until it's actually accessed.

```java
@OneToOne(fetch = FetchType.LAZY)
private EmployeeDetail employeeDetail;
```

&nbsp;

This is the **default** for `@OneToMany` and `@ManyToMany` relationships.

&nbsp;

```java
Employee employee = employeeRepository.findById(1L).get();
// employeeDetail is not loaded yet

// This will trigger a database query to load the employeeDetail
String address = employee.getEmployeeDetail().getAddress();
```
