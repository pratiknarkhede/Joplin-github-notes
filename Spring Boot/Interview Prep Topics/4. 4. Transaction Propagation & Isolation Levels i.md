#### **1\. Propagation (How Transactions Nest)**

**Key for Interviews**: Focus on `REQUIRED` and `REQUIRES_NEW`.

- **`REQUIRED` (Default)**
    
    - Joins existing transaction; creates new if none.
    - **When to use**: 90% of cases (e.g., service calling repository).
    
    ```java
    @Transactional  // Default is REQUIRED
    public void methodA() {
        methodB();  // Shares methodA's transaction.
    }
    ```
    
- **`REQUIRES_NEW`**
    
    - Suspends current transaction, starts a fresh one.
    - **When to use**: Audit logs, async tasks (must not roll back with main tx).
    
    ```java
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void auditLog() { /* New tx, independent of caller */ }
    ```
    

**Interview Q**:  
*"What happens if `REQUIRED` calls `REQUIRES_NEW`?"*

> *"`REQUIRES_NEW` pauses the caller‚Äôs tx, runs in a new tx, and resumes caller‚Äôs tx after. Rollbacks are independent."*

#### **2\. Isolation (How Txs See Each Other)**

**Key for Interviews**: Know `READ_COMMITTED` vs. `SERIALIZABLE`.

- **`READ_COMMITTED` (Default)**
    
    - No dirty reads (uncommitted changes invisible).
    - **Pitfall**: Non-repeatable reads (re-querying may show different data).
- **`SERIALIZABLE`**
    
    - No dirty/non-repeatable/phantom reads.
    - **Trade-off**: Slow (locks rows/tables).
    - **Use case**: Stock trading, banking.

**Interview Q**:  
*"When would you use `SERIALIZABLE`?"*

> *"Only for financial systems needing absolute consistency. Avoid due to performance cost."*

* * *

#### **3\. Rollback Rules**

- **Default**: **Only `RuntimeException` triggers rollback.**
    
- **Customize**:
    
    ```java
    @Transactional(rollbackFor = {SQLException.class})
    ```
    

**Trick Q**:  
*"If a checked exception is thrown but not in `rollbackFor`, does it roll back?"*

> *"No, unless it‚Äôs also a `RuntimeException`."*

#### **4\. Internal Method Call Trap**

**Problem**:

```java
@Transactional  
public void methodA() {
    methodB();  // ‚ùå No transaction! (Proxy bypassed)
}

@Transactional  
public void methodB() { /* ... */ }
```

**Fix**:

1.  **Self-injection**:
    
    ```java
    @Autowired private MyService self;  
    self.methodB();  // ‚úÖ Works (but hacky).  
    ```
    
2.  **`TransactionTemplate`**:
    
    ```java
    @Autowired TransactionTemplate txTemplate;  
    txTemplate.execute(() -> { methodB(); return null; });
    ```
    

**Interview Q**:  
*"Why doesn‚Äôt `@Transactional` work for internal calls?"*

> *"Spring proxies only intercept external calls. Internal calls bypass the proxy."*

### **Interview Hack**

If asked to **design a transactional service**:

```java
@Service
public class PaymentService {
    @Transactional  // REQUIRED by default
    public void processPayment() {
        deductAmount();
        updateLedger();  // Both in same tx.
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logAudit() { /* Always new tx */ }
}
```

**Mention**:

- *"I‚Äôd use `REQUIRES_NEW` for audit logs to ensure they persist even if payment fails."*
