When the Spring context contains multiple beans of the same type, the framework faces ambiguity when deciding which bean to inject.

#### **Scenario 1: Matching by Parameter Name**

- **Behavior**: Spring selects a bean whose name matches the parameter name in a constructor or method.
    
    ```java
    @Configuration
    public class ProjectConfig {
    
        @Bean
        public Parrot parrot1() {
            Parrot p = new Parrot();
            p.setName("Koko");
            return p;
        }
    
        @Bean
        public Parrot parrot2() {
            Parrot p = new Parrot();
            p.setName("Miki");
            return p;
        }
    
        @Bean
        public Person person(Parrot parrot2) {  // Matches the bean named 'parrot2'
            Person p = new Person();
            p.setName("Ella");
            p.setParrot(parrot2);
            return p;
        }
    }
    
    ```
    
    * * *
    
    #### **Using `@Primary`**
    
    - **Behavior**: When multiple beans exist, Spring selects the bean marked as primary.
        
        ```java
        @Bean
        @Primary
        public Parrot parrot2() {
            Parrot p = new Parrot();
            p.setName("Miki");
            return p;
        }
        
        ```
        
        **Effect**:
        
        - Even without specifying a parameter name, Spring defaults to the `@Primary` bean.

* * *

#### **Using `@Qualifier`**

- **Definition**: The `@Qualifier` annotation explicitly specifies which bean to inject, avoiding ambiguity.
    
    ```java
    @Configuration
    public class ProjectConfig {
    
        @Bean
        public Parrot parrot1() {
            Parrot p = new Parrot();
            p.setName("Koko");
            return p;
        }
    
        @Bean
        public Parrot parrot2() {
            Parrot p = new Parrot();
            p.setName("Miki");
            return p;
        }
    
        @Bean
        public Person person(@Qualifier("parrot2") Parrot parrot) {
            Person p = new Person();
            p.setName("Ella");
            p.setParrot(parrot);
            return p;
        }
    }
    
    ```
    

&nbsp;

* * *

#### **Scenario 4: No Matching Strategy**

- If none of the above strategies is applied:
    - **Result**: Spring throws a `NoUniqueBeanDefinitionException`.
    - **Solution**: Use `@Primary` or `@Qualifier` to resolve the conflict.

* * *

### **Key Takeaways**

1.  When multiple beans of the same type exist, Spring needs a strategy to resolve ambiguity.
2.  Use **parameter names**, **@Primary**, or **@Qualifier** annotations to guide Spring in selecting the correct bean.
3.  The `@Qualifier` annotation is explicit and robust, reducing the risk of issues during refactoring.
4.  Failing to specify a bean in ambiguous situations results in a `NoUniqueBeanDefinitionException`.
