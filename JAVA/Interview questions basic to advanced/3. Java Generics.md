### **What Problem Does Generics Solve?**

- **Type Safety**: Generics ensure that you can work with collections and other types without `ClassCastException`. It enforces type checks at compile time.
- **Code Reusability**: Allows code to work with different types while avoiding code duplication.
- **Elimination of Casts**: With generics, explicit casting is not required when retrieving elements from a collection.

* * *

### **Generics Resolution at Compile Time (Type Erasure)**

- **Erasure**: Generics are resolved at compile time by the Java compiler. After compilation, the generic types are **erased**, and the code runs as if it were written without generics.
- **Type Erasure Steps**:
    1.  **Type Parameters are replaced** by their bounds or `Object` if unbounded.
    2.  **Cast Insertions**: The compiler inserts necessary type casts.
    3.  **Bridge Methods**: The compiler creates bridge methods to preserve polymorphism during type erasure.

```java
class Box<T> {
    private T value;
    public T getValue() { return value; }
}

//After type erasure:


class Box {
    private Object value;
    public Object getValue() { return value; }
}

```

### **Generics Cheatsheet**

| **Concept** | **Syntax** | **How to Read** | **Example** | **Comments** |
| --- | --- | --- | --- | --- |
| **Generic Class** | `class ClassName<T> {}` | "Class with type parameter T" | `class Box<T> {}` | Enables defining a class that can operate on objects of any type. |
| **Generic Method** | `<T> void method(T param)` | "Method with type parameter T" | `<T> void print(T param)` | Allows methods to accept a parameter of any type. |
| **Bounded Type** | `class ClassName<T extends SuperType>` | "T must extend SuperType" | `class Box<T extends Number>` | Restricts the type to subclasses or a specified type. |
| **Multiple Bounds** | `class ClassName<T extends A & B>` | "T must extend A and implement B" | `class Manager<T extends Person & Comparable<T>>` | A type must extend a class and implement an interface. |
| **Wildcard (`?`)** | `List<?>` | "List of unknown type" | `void printList(List<?> list)` | Allows passing any type. Generally used for reading. |
| **Upper Bound** | `List<? extends T>` | "List of objects that extend T" | `List<? extends Number> list` | Restricts the unknown type to be a subclass of `T`. |
| **Lower Bound** | `List<? super T>` | "List of objects that are supertype of T" | `List<? super Integer> list` | Allows `Integer` or its supertype, ensuring write-safety for generic collections. |
| **Generic Interface** | `interface InterfaceName<T>` | "Interface with type parameter T" | `interface Container<T>` | Allows defining an interface with type parameters. |
| **Diamond Operator** | `new ClassName<>()` | "Inferred generic type" | `List<String> list = new ArrayList<>();` | The diamond operator infers the generic type from the context, reducing verbosity. |
| **Type Erasure** | `class ClassName<T>` -> `class ClassName` | "Generics erased to raw type" | `class Box<T>` becomes `class Box` after compile | Generics only exist during compilation; no generic info at runtime. |
| **Wildcard Capture** | `<?>` -> `Object` | "Unknown type captured as Object" | `void method(List<?> list)` -> internally Object | Type inference converts unknown types to `Object` within method body. |
| **Raw Type** | `ClassName` | "Class with no generic parameters" | `List list = new ArrayList();` | Raw types disable type safety, reintroducing risks of `ClassCastException`. |

* * *

### **Key Points to Remember**

- **Generics at Runtime**: Generics are **compile-time constructs**, and the **JVM doesn't know** the actual generic types at runtime.
- **Type Bounds**: `T extends Class` restricts to subclasses of `Class`. Use `&` for multiple bounds.
- **Wildcard Types**: Use `? extends T` for covariance (read-only collections) and `? super T` for contravariance (write operations).
- **Type Safety with Wildcards**: Upper-bounded wildcards (`<? extends T>`) can be used for safe reading, but for adding elements, lower-bounded (`<? super T>`) should be used.

* * *

### **Wildcard Use Cases**

- **Reading data**: Use `<? extends T>`

```java
public void process(List<? extends Number> list) {
    // Can read as Number or its subclasses
}

```

- **Writing data**: Use `<? super T>`
    
    ```java
    public void insert(List<? super Integer> list) {
        list.add(10); // Can add Integer or its subclass
    }
    
    ```
    

* * *

- **Generics and Reflection**: Since generics are erased, you cannot get the actual generic type through reflection at runtime.
- **Overloading with Generics**: You cannot overload methods that only differ by their generic type parameters, as they are erased during compilation.
- **Instanceof with Generics**: You cannot use `instanceof` with parameterized types. For example, `if (obj instanceof List<String>)` is not valid; instead, use `List<?>`.
