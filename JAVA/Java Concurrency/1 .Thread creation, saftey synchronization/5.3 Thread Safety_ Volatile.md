## Volatile Keyword

The `volatile` keyword ==ensures that a variable's value is always read from and written to main memory, not from CPU caches.==

&nbsp;

### Visibility vs. Atomicity

- **Visibility**: When one thread modifies a variable, other threads can see that change
- **Atomicity**: An operation completes in a single, uninterruptible step

&nbsp;

### When to Use Volatile

- For variables accessed by multiple threads
- When you need visibility guarantees but don't need atomic compound operations
- For flags that can terminate threads

&nbsp;

The volatile keyword ensures visibility of variable changes across threads, preventing caching issues.

Means it

&nbsp;

```java
package com.pratik.thejavajourney.concurrency.printInorder.thread_creation;

public class VolatileUsecase {
    private volatile boolean running=true;
    public void stop(){
        running=false;
    }

    public void run() {
        while(running){
            System.out.println("Thread is still running..");
        }
        try {
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
        System.out.println("thread stopped execution ");
    }

    public static void main(String[] args) throws InterruptedException {
        VolatileUsecase volatileUsecase=new VolatileUsecase();
        Thread t=new Thread(volatileUsecase::run);
        t.start();
        Thread.sleep(1000);
        volatileUsecase.stop();
        t.join();
    }

}

```

&nbsp;

### What Volatile Doesn't Do

Volatile ensures visibility but doesn't make compound operations atomic. Consider:

```java
volatile int counter = 0;
counter++; // Still not thread-safe because it's a read-modify-write operation
```

&nbsp;

## Thread Safety Comparison

| Mechanism | Use When | Limitations |
| --- | --- | --- |
| Immutable Objects | You need objects that never change after creation | May create many objects if frequent updates needed |
| Atomic Classes | You need simple variables with atomic operations | Limited to specific operations (get, set, increment, etc.) |
| Volatile | You need visibility guarantees between threads | Doesn't provide atomicity for compound operations |
